Fundamental concepts and assumptions

The smart transport of product data needs a clear and common understanding of the format to express the data of the products itself. The product data can be transferred in several ways, e.g. via direct database connect, or via webservices or file transfer. In all cases, the sender and the receiver of the product data need a basic standard to implement the data transfer.

Normally an IFC files is known as a structure to transport a data model of a construction like a building (or a bridge). This model contains the geometrical and alphanumeric information of the complete building. The model contains also the information about the products that should be used to build up the building. The IFC language is capable to store the information of this “built-in” products properly.

This standard uses the IFC language to store and transport product data on the way from the manufacturers to their customers.

To store product data in an IFC file only some parts of the existing IFC namespace is needed, since many aspects of the BIM data are not needed to express product data. The subset of IFC structures for storing and transporting product data is described in the Model View Definition (MVD) \\\\\\\\\\\\*\\\\\\\\\\\\*“Product Data View”\\\\\\\\\\\\*\\\\\\\\\\\\*(PDV).\\\\\\\\\\\\*\\\\\\\\\\\\*\\\\\\\\\\\\*\\\\\\\\\\\\*

The following sections focusses on a definition of a file format. This file can be the foundation to transfer data over different transport channels.

The file is based on IFC (Industry Foundation Classes). IFC was developed by the worldwide organization buildingSMART. IFC is standardized as \\\\\\\\\\\\*\\\\\\\\\\\\*ISO 16739\\\\\\\\\\\\*\\\\\\\\\\\\* and as \\\\\\\\\\\\*\\\\\\\\\\\\*EN ISO 16739\\\\\\\\\\\\*\\\\\\\\\\\\*.

Product data contains three types of data:

  1. Alphanumeric Information (Lists)
  2. Information in Documents (Files)
  3. Geometric Information (Visual Representation)

The existing proposals for new work items in CEN/TC442-WG4 describe the needs for transporting alphanumeric information. We propose to consider also the transport of Files. The transport of geometric information can be realized by including existing files with the geometric representation as a file in the IFCZIP-Container. With IFC this could be extended in a next step to an intelligent connected geometric representation.

The product data file is specified as follows:

Standard EN ISO 16739
Language IFC (Industry Foundation Classes)
Language Version IFC 4, ADD 2
Language Type ifcXML
File Type ifcZIP

The IFC language contains hundreds of entities to provide a standardizes data exchange for the whole construction industry. For exchanging product data based on the specification of this paper, only some of these entities are needed. These entities are defined in this specification. The IFC specification is located here: http://www.buildingsmart-tech.org . All elements are well documented there. In this paper, we describe the use of the elements and attributes for the use case of transporting product data templates (PDT) and corresponding product data (PD).

The product data shall be transportable from the sender (e.g. the manufacturer) to the receiver (e.g. customer). The product data templates (PDT) is an empty form. The receiver can hand the empty form over to the sender, to make clear, which information shall be delivered. The sender then fills the information into the form and gives it back to the receiver.
