1.1 Construction Objects Data View

COD.10.9CandidateCEN-TC442-WG2-TG3CEN - European Committee for Standardization

The scope of this standard is to define the syntactic characteristics of a generic document produced to carry data and information basing on EN ISO 16739, EN ISO 23386 and EN ISO 23387 standards. The work has been also designed in accordance with other ongoing CEN TC442 WG4 works such as N17_NWIP 1: Properties - Part 1 Description methodology for BIM and other digital processes, N19_NWIP 3: Part 1 General structure of a product data template, N20_NWIP 4: Part 2 Framework for product data templates under CPR.

The application of such standards will allow manufacturers to structure, govern, update, maintain and transport their product information through a machine-readable format. In addition, several stakeholders of the process will be able to take the most out from their software technologies, avoiding data replication as well as saving time and improving both quality and transparency. The scenario within which this standard will apply will affects in fact a wide spectrum of professionalism: from producers to designers, from constructors to facility and property managers.

The syntactic approach considers as primary objective the ability to re-use specifications given in the international standard EN ISO 16739. This specific goal is going to be reached by the delivery of a new Model View Definition, as intended at buildingSMART.org, named “Construction Objects Data View” which will be made available alongside the others MVD already issued. Software houses and software application developers will find this work use and implementation simple and profitable because based on a widely accepted standard like EN ISO 16739.

Because digital data transmittal needs agreed structured data, predefined schemas and agreed property definitions, this standard conceptually splits between Data Template and a Data Sheet. Data Template is going to be defined as the demanded data schema, still missing data but possibly including limitations or required ranges, while Data Sheet represents rather the answer including completed or partially filled data. The first might represent level of information need requirements and definitions, general requests, while the second might represent catalogs, offering of products, selected or filtered data, approved data and so on. Data Sheet may be used as the input to be added into informative modeling storages as well as information to be saved in related repositories as Common Data Environments or Asset Information Models.

The work aims above all the scope of filling the technological gap still today preventing an effective digital dialogue between production and construction chains. Unfortunately, such a dialog is traditionally based on documentation, no matter if printed or digital, which for the most cases is not digitally computable. A secondary work objective is to constitute itself as a syntax suitable for the digital management of the contents requested by the Construction Products Regulation (CPR) which lays down harmonized rules for the distribution of construction products in the EU market, particularly covering demand of comparison of products performance, characteristics, certification, CE marking and conformity declarations. 

The smart transport of product data needs a clear and common understanding of the format to express the data of the products itself. The product data can be transferred in several ways, e.g. via direct database connect, or via webservices or file transfer. In all cases, the sender and the receiver of the product data need a basic standard to implement the data transfer.

Normally an IFC files is known as a structure to transport a data model of a construction like a building (or a bridge). This model contains the geometrical and alphanumeric information of the complete building. The model contains also the information about the products that should be used to build up the building. The IFC language is capable to store the information of this “built-in” products properly.

This standard uses the IFC language to store and transport product data on the way from the manufacturers to their customers.

To store product data in an IFC file only some parts of the existing IFC namespace is needed, since many aspects of the BIM data are not needed to express product data. The subset of IFC structures for storing and transporting product data is described in the Construction Objects Data View (COD).

The following sections focusses on a definition of a file format. This file can be the foundation to transfer data over different transport channels.

The file is based on IFC (Industry Foundation Classes). IFC was developed by the worldwide organization buildingSMART. IFC is standardized as EN ISO 16739.1:2018.

Product data contains three types of data:

  1. Alphanumeric Information (Lists)
  2. Information in Documents (Files)
  3. Geometric Information (Visual Representation)

The existing proposals for new work items in CEN/TC442-WG4 describe the needs for transporting alphanumeric information. We propose to consider also the transport of Files. The transport of geometric information can be realized by including existing files with the geometric representation as a file in the IFCZIP-Container. With IFC this could be extended in a next step to an intelligent connected geometric representation.

The IFC language contains hundreds of entities to provide a standardizes data exchange for the whole construction industry. For exchanging product data based on the specification of this paper, only some of these entities are needed. These entities are defined in this specification. The IFC specification is located here: https://technical.buildingsmart.org . All elements are well documented there. In this paper, we describe the use of the elements and attributes for the use case of transporting construction objects data templates (DT) and corresponding construction objects data (PD).

The product data shall be transportable from the sender (e.g. the manufacturer) to the receiver (e.g. customer). The construction objects data templates (DT) is an empty form. The receiver can hand the empty form over to the sender, to make clear, which information shall be delivered. The sender then fills the information into the form and gives it back to the receiver.

Relation of part 1 and part 2

The relation of this standard (Part1) the Part 2 is as following:

• The use case "Detailed description of a product (pre-design programs, technical specifications, offers)" will be covered by Part 1 and Part 2 together • The NWIP shall make clear, that there is also a Part 1 and both parts are needed to support all use cases (Part 1 and Part 2) • The part 2 will define the way, how a (requirements) request can be formulated with ISO 16739. These requests shall be applicable to product libraries based on Part 1. • The part 2 will define a way, to create a constraint-based product library for highly configurable products to support an interactive selection process. The application of a request to a "Part1-Library" and to a "Part2-Library" shall response same products as the result. • The name "Product Library" is kind of a synonym to the term "product catalogue". • A "Part1-Library" can be automatically transformed to a "Part2-Library" and vice versa; so, these can be seen as different representations of the same content. This does not mean, that such a transformation (esp. from P1=>P2) includes all rules a product can offer, it would just include those rules, that can be extracted from a variant-based product-list from P1. • It is up to the manufacturers to choose the way, they want to represent their products ("Part1-Library" and/or "Part2-Library"), they just shall be sure, that all requests can be applied to both types of libraries. We think, that those manufacturers, who are willing to expose all configuration rule in the design process, will probably implement P2 in the end, but the implementation of P1 could be a first step, since some manufacturers have already share product lists with us, that can easily be express in the P1-way. So, it would be good to offer transformation paths from P1 to P2.

Exchange requirements

The standard contains four definitions of information exchange requirements.

Each element of the exchange requirements corresponds to an element in EN ISO 16739.1:2018. The documentation of EN ISO 16739.1:2018 (IFC 4, ADD 2, TC 1, Version 4.0.2.1) will not be repeated in this document.

The elements from ISO EN 16739.1:2018 that are applicable for the exchange requirements are defined in detail in the chapter Technical specification in the included HTML documentation.

Transport

All exchanges can be transported over these protocols:

  1. As a physical file with manual exchange via eMail, Harddrives, oder similar
  2. As a data structure in the payload of an online exchange, eg. an API, a messaging system or similar.

This standard does not define how exactly the exchange is done. It defines the standard for the structure of the file or the payload. The file or the payload can be stored in a compressd format according to the specification. The file extension is then \\\\\*.ifczip. The specifications for ifczip can be found here:

• http://www.buildingsmart-tech.org/specifications/ifc-overview/ifc-overview-summary • http://www.buildingsmart-tech.org/implementation/ifc-implementation/ifc-impl-agreements/cv-2x3-154

It is explicitly allowed to embed an IFC file in Step format (.ifc) or in XML format Format (.ifcxml) in an ifczip file. The specification contains the following principles:

  1. The ifczip file contains exactly one model file (.ifc or .ifcxml). The file extension is \\\\\*.ifczip
  2. According to this rule, it is allowed to embed additional other files in an ifczip file. Therefore, an ifczip file can be a container, which can transport many additional files in itself.
  3. Such a container offers several advantages, such as • transport of photo documentations • transport of plant documentation • transport of permits and notifications • transport of operating documents, e.g. test and maintenance reports

In the BIM context, such a transport is particularly useful if the documents have a reference to the structure.

In order to be able to exchange data without consultation, each transported document should be registered in a structured, electronic table of contents. The recipient of the container can first read the table of contents and then select and remove the files.

For this purpose, two things must be specified for the sender and receiver of such a container file: • syntax of the table of contents • specifications for the semantic identification of documents Since the current specification of the ifczip format provides exactly one model file in the root folder of the ifczip file, it makes sense to save the table of contents and the catalogue of document types in this model file. For this purpose, the IFC data model (based on IFC4) provides the entity IfcDocumentInformation. For each file transported in the container, an element of type IfcDocumentInformation is written.
