4.5 Transport

All exchanges can be transported over these protocols:

  1. As a physical file with manual exchange via eMail, Harddrives, oder similar
  2. As a data structure in the payload of an online exchange, eg. an API, a messaging system or similar.

This standard does not define how exactly the exchange is done. It defines the standard for the structure of the file or the payload. The file or the payload can be stored in a compressd format according to the specification. The file extension is then \\\\\*.ifczip. The specifications for ifczip can be found here:

• http://www.buildingsmart-tech.org/specifications/ifc-overview/ifc-overview-summary • http://www.buildingsmart-tech.org/implementation/ifc-implementation/ifc-impl-agreements/cv-2x3-154

It is explicitly allowed to embed an IFC file in Step format (.ifc) or in XML format Format (.ifcxml) in an ifczip file. The specification contains the following principles:

  1. The ifczip file contains exactly one model file (.ifc or .ifcxml). The file extension is \\\\\*.ifczip
  2. According to this rule, it is allowed to embed additional other files in an ifczip file. Therefore, an ifczip file can be a container, which can transport many additional files in itself.
  3. Such a container offers several advantages, such as • transport of photo documentations • transport of plant documentation • transport of permits and notifications • transport of operating documents, e.g. test and maintenance reports

In the BIM context, such a transport is particularly useful if the documents have a reference to the structure.

In order to be able to exchange data without consultation, each transported document should be registered in a structured, electronic table of contents. The recipient of the container can first read the table of contents and then select and remove the files.

For this purpose, two things must be specified for the sender and receiver of such a container file: • syntax of the table of contents • specifications for the semantic identification of documents Since the current specification of the ifczip format provides exactly one model file in the root folder of the ifczip file, it makes sense to save the table of contents and the catalogue of document types in this model file. For this purpose, the IFC data model (based on IFC4) provides the entity IfcDocumentInformation. For each file transported in the container, an element of type IfcDocumentInformation is written.
